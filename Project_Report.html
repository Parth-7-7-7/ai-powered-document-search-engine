AI-Powered Legal Document Search Engine

Hybrid search combining BM25 keyword matching and semantic vector search via Reciprocal Rank Fusion

Trilegal Assignment Parth Rathi

1. Project Overview & Problem Statement

Legal professionals spend a significant amount of time searching through large collections of contracts and court judgments to find relevant clauses, precedents, or obligations. Traditional search tools rely only on exact keyword matching — if you search for "when can a party exit without being liable", a keyword search returns nothing because those exact words don't appear in the document, even though every termination clause answers that question.

This project solves that problem by building a hybrid search system that understands both exact words and the meaning behind a query. It runs entirely on a local machine with no internet connection needed at query time, no paid APIs, and no external dependencies beyond the initial setup.

Core problem: Keyword search misses conceptually relevant documents. Semantic search sometimes misses exact legal terms. Neither alone is sufficient for legal document retrieval. Hybrid search combining both — with intelligent fusion — performs better than either method independently.

The system supports three search modes displayed side-by-side: Keyword (BM25) for exact term matching, Semantic (MiniLM) for meaning-based retrieval, and Hybrid (RRF) which fuses both intelligently. A built-in "Difference Explainer" panel shows exactly which documents each method found and why they differ.

3. Technology Choices & Reasoning

Component Technology Used Why This Choice
Keyword Search BM25Okapi
(rank-bm25 library)		 BM25 is the industry standard for keyword retrieval. It is better than basic word counting (TF-IDF) because it handles two problems automatically: (1) a word appearing 100 times in a document shouldn't count 100× more than appearing 10 times — BM25 caps this; (2) a long document shouldn't rank higher just because it has more words — BM25 normalises for length. Works without any training data.
Embeddings all-MiniLM-L6-v2
(sentence-transformers)		 Converts text into 384-dimensional vectors where similar meaning = similar direction. Chosen because: (1) only 80 MB in size — fast to load; (2) runs fully offline — no API key or cost; (3) takes ~60 seconds to encode all 3,532 chunks on a CPU — acceptable for one-time ingestion; (4) well-known benchmark performance on semantic similarity tasks. A larger model would be more accurate but impractical without a GPU.
Vector Store FAISS IndexFlatIP
(faiss-cpu)		 FAISS (Facebook AI Similarity Search) is the go-to library for searching through large collections of vectors. We used IndexFlatIP (Inner Product) with L2-normalised embeddings, which gives exact cosine similarity. "Exact" means no approximation errors — every search finds the true nearest neighbour. At 3,532 vectors this is fast enough. At millions of vectors, we'd switch to FAISS HNSW which approximates but is much faster.
Hybrid Fusion Reciprocal Rank Fusion (RRF)
(custom implementation)		 BM25 scores and semantic cosine scores are on completely different scales — you cannot simply add them together. RRF solves this by working with ranks instead of scores. Formula: 1÷(60+rank_bm25) + 1÷(60+rank_sem). A document ranked consistently in both lists scores higher than one that is #1 in only one list. The constant 60 is the standard value from the paper that introduced RRF (Cormack et al., 2009).
Text Chunking RecursiveCharacterTextSplitter
(langchain-text-splitters)		 Legal documents can be 50,000+ characters long. We can't embed an entire document as one vector — it loses too much detail. Splitting into 500-character chunks with 50-character overlap means each chunk is specific enough to match a precise query, while the overlap prevents a clause from being cut in half between two chunks.
UI Streamlit Allows building an interactive web app in pure Python — no HTML/JavaScript needed. @st.cache_resource loads the search engine once and keeps it in memory across queries, avoiding reload overhead.
Key improvement made during testing: When a user types a natural language question (e.g., "what happens when a disaster makes performance impossible?"), BM25 matches on random tokens like "natural" and "government" in unrelated court judgments, contaminating hybrid results. We added a query-type detector that checks the ratio of stopwords in the query. If >30% of the words are stopwords (a, the, when, between, etc.) and the query is 6+ words long, BM25 is excluded from RRF and hybrid becomes semantic-only. This fixed contamination in 3 out of 10 test queries with zero regression on keyword queries.

4. Dataset

All 35 documents were downloaded from HuggingFace — a free, public platform where researchers share datasets. Two separate datasets were used:

Category Count Source Document Types
Commercial Contracts 20 CUAD v1 — The Atticus Project
dvgodoy/CUAD_v1_Contract_Understanding_PDF		 Supply, franchise, software license, co-branding, manufacturing, distributor, service, sponsorship agreements
Indian SC Judgments 15 rishiai/indian-court-judgements-and-its-summaries Income tax disputes, property inheritance, Hindu succession law, civil procedure, constitutional rights
Total 35 3,532 indexed chunks after splitting

The two document types were chosen deliberately — they use completely different vocabulary. Contracts use terms like indemnification, sublicense, force majeure. Judgments use terms like reversioner, estoppel, limitation act. This makes the search system harder to fool and more interesting to test across both keyword and semantic queries.

A script (collect_documents.py) handles the download automatically, saves each document as a plain text file, and creates a metadata.json index recording the filename, title, type, and source of every document. If the primary judgment dataset fails to download, the script automatically tries a backup dataset.

6. Assumptions & Limitations

Assumptions

Assumption Reason
35 documents are representative of the legal domain Covers two distinct legal areas (commercial contracts + Indian SC judgments) with sufficiently varied terminology to test all search modes meaningfully.
500-character chunk size is appropriate Legal clauses are typically self-contained within a paragraph. Smaller chunks lose context; larger chunks dilute precision of semantic matching.
Queries with >30% stopwords are natural language Empirically validated across all test queries — keyword queries consistently scored below 20%, sentence queries consistently scored above 35%.
MiniLM-L6-v2 embeddings are sufficient for this corpus The model performs well on general English text. For a 35-document corpus without GPU, it is the best balance of accuracy and speed.
No user authentication or access control needed System is designed as a local research tool, not a multi-user production service.

Limitations

Limitation Impact Production Fix
FAISS exact search (IndexFlatIP) Fine at 3,532 vectors, slow at millions Replace with FAISS HNSW or IVF approximate index
No cross-encoder reranking Retrieval quality limited to bi-encoder (MiniLM) precision Add a cross-encoder reranker after RRF
General-purpose embeddings MiniLM not fine-tuned on Indian legal text Fine-tune on Indian SC judgments or use InLegalBERT
Static index Adding documents requires full re-ingestion (~3 min) Switch to incremental FAISS + BM25 updates
Stopword-ratio heuristic Query-type detection can misclassify edge cases Replace with a trained query classifier
